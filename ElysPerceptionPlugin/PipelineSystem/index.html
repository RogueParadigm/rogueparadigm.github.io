Pipeline System

Deep dive into the ElysPerceptionPlugin's modular Pipeline architecture.

Overview​

The Pipeline is a modular, data-driven architecture for evaluating perception candidates. Instead of hardcoded perception logic, you configure a series of stages that can be mixed and matched.

Why Pipeline Architecture?​

Traditional perception problems:

  • Monolithic code that's hard to customize
  • Mixing concerns (sampling, filtering, scoring)
  • One-size-fits-all approach

Pipeline benefits:

  • Separation of concerns (each stage has one job)
  • Composability (mix default and custom components)
  • Reusability (same components across channels)
  • Extensibility (add custom logic without touching core)
  • Designer-friendly configuration

Core Philosophy​

  1. Data Over Code: Behavior is configured, not coded
  2. Immutable Context: All stages read from same snapshot
  3. Pure Functions: Stages evaluate without modifying state
  4. Independence: Multiple channels run in parallel
  5. Opt-In: Targets choose to participate via interfaces

Pipeline Flow​

Every tick, the perception component:

  1. Builds Context - Snapshot of player position, view direction, viewport info
  2. Runs Each Channel - Target, Interaction, custom channels independently
  3. Determines Focus - Which channel's result gets priority
  4. Fires Events - Notifies your game of changes

Execution Flow​

UERPPerceptionComponent::TickComponent()
  ↓
BuildContext()
  → Get owner actor
  → Get view point (position, rotation)
  → Extract aim ray
  → Calculate viewport data
  → Create FERPPerceptionContext
  ↓
For each channel:
  ↓
  Execute Pipeline:
    → Sampler: Gather candidates
    → Filters: Reject invalid
    → Scorers: Assign scores
    → Aggregator: Combine scores
    → Resolver: Select best
  ↓
  Store result (best candidate + score)
  ↓
Determine Focus Winner
Fire Events (OnCandidateAcquired/Lost)
Update Debug Visualization

Pipeline Stages​

Stage 1: Sampler​

Purpose: Collect potential candidate actors from the world.

Input:

  • FERPPerceptionContext - Perception context
  • float Range - Maximum detection range

Output:

  • TArray<AActor*> - All candidate actors

Signature:

void Sample(const FERPPerceptionContext& Context, float Range, TArray<AActor*>& OutCandidates) const;

Responsibility:

  • Query world for actors (overlap, ray cast)
  • Return raw list without filtering
  • Should be fast and broad

Default Implementation:

  • UERPSphereOverlapSampler - Sphere overlap query

Example Configuration:

Sampler: ERPSphereOverlapSampler
  - Trace Channel: Visibility
  - Actor Class: Character
  - Ignore Context Actor: ✓

Stage 2: Filters​

Purpose: Eliminate invalid candidates. ALL filters must pass.

Input:

  • FERPPerceptionContext - Perception context
  • AActor* Candidate - Single candidate to test

Output:

  • bool - True if passes, false if rejected

Signature:

bool Passes(const FERPPerceptionContext& Context, AActor* Candidate) const;

Responsibility:

  • Fast rejection of invalid candidates
  • Binary decision (pass/fail)
  • No scoring, just validation

Default Implementations:

  • UERPTargetableFilter - Requires IERPTargetable interface
  • UERPInteractableFilter - Requires IERPInteractable interface
  • UERPDistanceFilter - Range-based filtering

Execution: Filters run in sequence with short-circuit evaluation. If any filter returns false, candidate is rejected immediately.

Example Logic:

  • Check if actor implements required interface
  • Verify actor is alive
  • Check line of sight
  • Range validation

Stage 3: Scorers​

Purpose: Assign numerical scores to candidates. Multiple scorers evaluate different aspects.

Input:

  • FERPPerceptionContext - Perception context
  • AActor* Candidate - Candidate to score

Output:

  • float - Normalized score (typically 0.0-1.0)

Signature:

float Score(const FERPPerceptionContext& Context, AActor* Candidate) const;

Responsibility:

  • Evaluate one aspect of candidate
  • Return normalized score where possible
  • Should not eliminate candidates (use Filters)

Default Implementations:

  • UERPDistanceScorer - Scores by distance (lower = closer = better)
  • UERPConeScorer - Scores by angle from forward
  • UERPViewportEllipseScorer - Scores by screen position

Execution: All scorers run for each candidate, producing an array of scores per candidate.

Score Convention:

  • Lower scores are better (by convention)
  • Typically normalized to 0.0-1.0
  • Ensure all scorers share same convention

Example:

Candidate A: Distance 500 → score 0.25
Candidate B: Distance 1000 → score 0.5
Result: A wins (0.25 < 0.5)

Stage 4: Aggregator​

Purpose: Combine multiple scores into single final score per candidate.

Input:

  • TArray<float> Scores - All scores for one candidate

Output:

  • float - Single aggregated score

Signature:

float Aggregate(const TArray<float>& Scores) const;

Responsibility:

  • Reduce multiple scores to one value
  • Handle empty arrays
  • Define how scores combine

Default Implementations:

  • UERPIdentityAggregator - Returns first score (for single scorer)
  • UERPAdditiveAggregator - Sums all scores
  • UERPMultiplicativeAggregator - Multiplies all scores

Example:

Candidate A scores: [0.3 (distance), 0.2 (angle)]
Candidate B scores: [0.5 (distance), 0.1 (angle)]

Additive Aggregator:
  A: 0.3 + 0.2 = 0.5
  B: 0.5 + 0.1 = 0.6
  → A wins (lower score)

Stage 5: Resolver​

Purpose: Select the winning candidate from all scored candidates.

Input:

  • TArray<FERPPipelineScoredCandidate> - All candidates with scores

Output:

  • bool - True if winner found
  • FERPPipelineScoredCandidate& OutBest - The winner

Signature:

bool Resolve(const TArray<FERPPipelineScoredCandidate>& ScoredCandidates, FERPPipelineScoredCandidate& OutBest) const;

Responsibility:

  • Determine which candidate wins
  • Handle empty arrays (return false)
  • Define "better" (lowest vs highest)

Default Implementations:

  • UERPLowestScoreResolver - Selects lowest score
  • UERPHighestScoreResolver - Selects highest score

Context System​

FERPPerceptionContext​

The context is an immutable snapshot of perception data, built once per tick and shared across all channels.

Purpose:

  • Provide consistent data to all pipeline stages
  • Avoid repeated queries (GetActorLocation, etc.)
  • Decouple from specific owner types

Contents:

struct FERPPerceptionContext
{
    AActor* ContextActor;        // Owner (player)
    FVector Origin;              // Player position
    FVector Forward;             // Facing direction
    
    // Optional aim ray
    bool bHasAimRay;
    FVector AimOrigin;
    FVector AimDirection;
    
    // Optional viewport projection
    bool bHasViewportProjection;
    FVector2D ViewportCenterPx;
    float ViewportEllipseHalfWidthPx;
    float ViewportEllipseHalfHeightPx;
};

Context Building​

  1. Get Owner Actor (component's owner)
  2. Get View Point (location and rotation)
  3. Extract Aim Ray (optional camera aim)
  4. Calculate Viewport Data (for PlayerController)

Why Immutable?​

Benefits:

  • Predictable behavior across stages
  • No hidden state changes
  • Easy to debug and test
  • Thread-safe (future)

Implications:

  • Pipeline stages cannot modify context
  • All stages see identical data
  • Context rebuilt each tick

Multi-Channel Support​

Channel Independence​

Each channel runs its own complete pipeline:

  • Own sampler, filters, scorers, aggregator, resolver
  • Own best candidate
  • Own events
  • Independent configuration

Example:

Target Channel:
  - Range: 3000
  - Filters: TargetableFilter
  - Scorers: DistanceScorer, ConeScorer
  
Interaction Channel:
  - Range: 300
  - Filters: InteractableFilter
  - Scorers: DistanceScorer

Shared Context​

While channels are independent, they share the same context:

  • Same origin point
  • Same forward direction
  • Same viewport data
  • Built once, used by all

Adding Custom Channels​

You can add channels beyond Target and Interaction:

UPROPERTY(EditAnywhere, BlueprintReadWrite, Category = "Perception|Pipelines")
FERPChannelPipeline CustomPipeline;

Typical setups use 2-3 channels for performance.

Performance​

Per Tick Cost​

  • 1x Context build (cheap)
  • N x Pipeline executions (one per channel)
  • M x Candidate evaluations (depends on sampler)

Optimization Tips​

Reduce Sampling Range: Keep DefaultSamplingRange reasonable Use Filters Early: Reject candidates before expensive scoring Lightweight Scorers: Keep scoring logic simple Adjust Tick Rate: Use PrimaryComponentTick.TickInterval if real-time not needed
