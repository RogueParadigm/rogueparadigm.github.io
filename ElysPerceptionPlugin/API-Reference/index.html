Multi-Channel Architecture‚Äã

ElysPerceptionPlugin uses a pure perception, multi-channel architecture:

  • Channels: Independent evaluation pipelines (e.g., "Target", "Interaction", custom channels)
  • Pure Perception: The component only identifies candidates; it does NOT make gameplay decisions, manage focus, or prioritize channels
  • No Cross-Channel Logic: Each channel evaluates independently and stores its best candidate
  • Listener Components: Use UERPChannelListenerComponent to subscribe to specific channels and respond to acquisition/loss events

Key Design Principles‚Äã

  1. No Winner/Focus/Lock: The perception component does not decide which channel "wins" or manage any locking mechanism
  2. Channel-Based Events: Generic events fire per-channel (OnChannelCandidateAcquired, OnChannelCandidateLost)
  3. Gameplay-Agnostic: All gameplay semantics (targeting, interaction, UI) are handled outside the perception system
  4. Flexible Listening: Use multiple UERPChannelListenerComponent instances to subscribe to different channels independently

Interfaces‚Äã

IERPTargetable‚Äã

Query-only interface for actors that can be targeted.

Header: Core/ERPTargetable.h

UINTERFACE(BlueprintType)
class UERPTargetable : public UInterface
{
    GENERATED_BODY()
};

class IERPTargetable
{
    GENERATED_BODY()

public:
    /**
     * Returns whether this Actor currently accepts being targeted by the specified Actor.
     * 
     * @param TargetingActor The Actor attempting to target this Actor.
     */
    UFUNCTION(BlueprintCallable, BlueprintNativeEvent, Category = "Targeting|Permissions")
    bool CanBeTargetedBy(AActor* TargetingActor) const;
};

Contract:

  • Answers: "Does this Actor currently accept being targeted by the given Actor?"
  • This interface must NOT drive gameplay, UI, input, or targeting workflows
  • Used by perception pipeline filters (e.g., UERPTargetableFilter)

Usage:

Blueprint:

  1. Class Settings ‚Üí Interfaces ‚Üí Add ERPTargetable
  2. Implement Can Be Targeted By event/function
  3. Return true to allow targeting, false to reject

C++:

class AMyEnemy : public ACharacter, public IERPTargetable
{
    virtual bool CanBeTargetedBy_Implementation(AActor* TargetingActor) const override
    {
        // Only allow targeting if alive and not on same team
        return bIsAlive && !IsSameTeam(TargetingActor);
    }
};

Scorers‚Äã

Score rejection convention

Some scorers may reject a candidate by returning ERPPipelineScore::RejectScore. This value is defined in Pipeline/ERPPipelineScoreConstants.h and is interpreted by the pipeline runner as a hard reject.

UERPDistanceScorer‚Äã

Scores candidates based on distance from origin.

Header: Pipeline/Scorers/ERPDistanceScorer.h

PropertyTypeDefaultDescription
bNormalizeByRangebooltrueNormalize distance to 0-1 range
MaxDistancefloat1000.0Maximum distance for normalization

Algorithm:

Distance = |Candidate.Location - Context.Origin|
If (bNormalizeByRange):
    Return Clamp(Distance / MaxDistance, 0..1)
Else:
    Return Distance

If Context/Candidate invalid:
    Return ERPPipelineScore::RejectScore

Score Meaning:

  • Lower score = closer = better (use with LowestScoreResolver)
  • 0.0 = at origin
  • 1.0 = at/beyond MaxDistance (when normalized)

UERPConeScorer‚Äã

Scores candidates based on angle from forward vector.

Header: Pipeline/Scorers/ERPConeScorer.h

PropertyTypeDefaultDescription
ConeHalfAngleDegreesfloat10.0Half-angle of acceptance cone (0-89)
bRejectOutsideConebooltrueReject candidates outside cone

Algorithm:

Direction = Normalize(Candidate.Location - Context.Origin)
Dot = Dot(Direction, Context.Forward)

If (bRejectOutsideCone && Dot < Cos(ConeHalfAngle)):
    Return ERPPipelineScore::RejectScore

Return 1.0 - Dot  // 0 = centered, 1 = at cone edge

Score Meaning:

  • Lower score = more centered = better
  • 0.0 = directly ahead
  • 1.0 = at cone edge
  • RejectScore = rejected (outside cone when enabled)

UERPViewportEllipseScorer‚Äã

Scores candidates based on their screen-space position within an ellipse. Requires viewport info - only works client-side.

Header: Pipeline/Scorers/ERPViewportEllipseScorer.h

PropertyTypeDefaultDescription
bUseDefaultScreenReferencebooltrueUse context's screen reference, or custom center
ScreenCenterNdcFVector2D(0.5, 0.5)Custom ellipse center in NDC (0..1), used if bUseDefaultScreenReference is false
EllipseHalfSizeNdcFVector2D(0.05, 0.05)Ellipse half-size in NDC (0..1)
bRejectOutsideEllipsebooltrueReject candidates outside the ellipse

Setup Requirements:

  1. Set bProvideViewportInfoWhenAvailable = true on ERPPerceptionComponent
  2. Owner must be a Character with a PlayerController (local player)
  3. Only works in PIE/Standalone (client-side, requires viewport)

Algorithm:

1. Validate viewport info exists (Context.bHasViewport)
2. Determine ellipse center:
   - If bUseDefaultScreenReference: use Context.ScreenReferenceNdc
   - Else: use ScreenCenterNdc
3. Project candidate to screen using PlayerController
4. Calculate normalized distance from ellipse center
5. If outside ellipse and bRejectOutsideEllipse: return RejectScore
6. Return 1.0 - distance (closer = higher score)

Score Meaning:

  • 1.0 = at ellipse center
  • 0.0 = at ellipse boundary
  • Negative = outside (if not rejected)
  • RejectScore = invalid or rejected

Use Cases:

  • FPS/TPS reticle-based interaction
  • Screen-center target selection
  • UI-based object highlighting
  • Multiple screen zones (using bUseDefaultScreenReference=false)

Aggregators‚Äã

UERPIdentityAggregator‚Äã

Header: Pipeline/Aggregators/ERPIdentityAggregator.h

Algorithm:

If (Scores.Num() > 0):
    Return Scores[0]
Else:
    Return 0.0

Use Cases:

  • Single scorer pipelines
  • No score combination needed
  • Performance optimization

UERPAdditiveAggregator‚Äã

Sums all scores together.

Header: Pipeline/Aggregators/ERPAdditiveAggregator.h

Algorithm:

Sum = 0.0
For each Score in Scores:
    Sum += Score
Return Sum

Score Meaning:

  • Each scorer contributes additively
  • Distance penalty + angle penalty = total penalty

Use Cases:

  • Combining independent penalties
  • Accumulating multiple factors
  • Weighted sum (use weights in scorers)

Example:

DistanceScore = 0.5  (50% of max range)
AngleScore = 0.3     (30% off center)
Total = 0.8          (combined penalty)

UERPMultiplicativeAggregator‚Äã

Multiplies all scores together.

Header: Pipeline/Aggregators/ERPMultiplicativeAggregator.h

Algorithm:

Product = 1.0
For each Score in Scores:
    Product *= Score
Return Product

Score Meaning:

  • Each scorer acts as a multiplier/modifier
  • One zero score makes total zero
  • All scores affect final result

Use Cases:

  • Weighted modifiers
  • Confidence scaling
  • Probability combination

Example:

BaseScore = 0.5      (base priority)
VisibilityMod = 0.8  (80% visible)
AngleMod = 0.9       (good angle)
Total = 0.36         (0.5 * 0.8 * 0.9)

Resolvers‚Äã

UERPLowestScoreResolver‚Äã

Header: Pipeline/Resolvers/ERPLowestScoreResolver.h

Algorithm:

If (ScoredCandidates.Num() == 0):
    Return false

Best = ScoredCandidates[0]
For each Candidate in ScoredCandidates:
    If (Candidate.Score < Best.Score):
        Best = Candidate

OutBest = Best
Return true

Use Cases:

  • Distance-based targeting (closest wins)
  • Penalty-based scoring (lowest penalty wins)
  • Most default pipelines

UERPHighestScoreResolver‚Äã

Selects the candidate with the highest score.

Header: Pipeline/Resolvers/ERPHighestScoreResolver.h

Algorithm:

If (ScoredCandidates.Num() == 0):
    Return false

Best = ScoredCandidates[0]
For each Candidate in ScoredCandidates:
    If (Candidate.Score > Best.Score):
        Best = Candidate

OutBest = Best
Return true

Use Cases:

  • Priority-based targeting (highest priority wins)
  • Threat level scoring (most threatening wins)
  • Affinity systems (best match wins)

Pipeline.ChannelId = "Interaction";
Pipeline.Sampler = NewObject<UERPSphereOverlapSampler>();

Pipeline.Aggregator = NewObject<UERPIdentityAggregator>();

Multi-Scorer Pipeline‚Äã

    NewObject<UERPDistanceScorer>(),  // Factor 1: Distance

TargetListener->OnCandidateAcquired.AddDynamic(this, &AMyController::OnTargetAcquired);
TargetListener->OnCandidateLost.AddDynamic(this, &AMyController::OnTargetLost);

UERPChannelListenerComponent* InteractionListener = CreateDefaultSubobject<UERPChannelListenerComponent>(TEXT("InteractionListener"));
InteractionListener->ChannelId = "Interaction";
InteractionListener->OnCandidateAcquired.AddDynamic(this, &AMyController::OnInteractableFound);
InteractionListener->OnCandidateLost.AddDynamic(this, &AMyController::OnInteractableLost);
