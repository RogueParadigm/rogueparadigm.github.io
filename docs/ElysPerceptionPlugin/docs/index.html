Overviewâ€‹

ElysPerceptionPlugin gives you a data-driven perception system you can assemble from reusable pieces. Instead of hardcoding targeting/interaction logic, you plug together Pipelines that sample, filter, score, aggregate, and resolve candidates each tick.

At a glance:

  1. Build a FERPPerceptionContext (origin, forward, viewport infoâ€¦)
  2. Run one or more channels (Target, Interaction, custom) through Pipelines
  3. Get events when candidates are acquired or lost

What it includes

  • Configurable Pipelines (Samplers, Filters, Scorers, Aggregators, Resolvers)
  • Per-channel focus resolution (Target vs Interaction by default)
  • Blueprint & C++ friendly APIs
  • Debug drawing and validation helpers

What it does not include

  • No built-in AI behavior trees/blackboards
  • No UI widgets; you bind to events and drive your own UI

Key Conceptsâ€‹

  • UERPPerceptionComponent: attach to PlayerController, Pawn, or custom Actor; owns channels and Pipelines.
  • Channels: independent evaluation tracks (e.g., Target, Interaction). Each has its own Pipeline and winner.
  • Pipeline Stages:
    • Sampler: gathers candidate actors (e.g., sphere overlap, custom query)
    • Filters: reject invalid actors (interfaces, distance, custom rules)
    • Scorers: assign scores (lower is better by default)
    • Aggregator: combine multiple scorer outputs
    • Resolver: pick the winner candidate
  • Context (FERPPerceptionContext): origin/forward, aim ray, viewport projection info, and owning actor. Used by Samplers/Filters/Scorers.
  • Interfaces:
    • IERPTargetable: opt-in for targeting
    • IERPInteractable: opt-in for interaction
    • IERPPerceptionContextProvider: customize context building

Quick Startâ€‹

Minimal setup (full guide in GettingStarted):

  1. Add component: ERPPerceptionComponent on your PlayerController or Pawn.
  2. Configure Target Pipeline (component details):
    • Channel Id: Target
    • Sampler: ERPSphereOverlapSampler (TraceChannel: Visibility, ignore context actor)
    • Filter: ERPTargetableFilter
    • Scorer: ERPDistanceScorer (normalize by range)
    • Aggregator: ERPIdentityAggregator (pass-through)
    • Resolver: ERPLowestScoreResolver
  3. Make actors targetable: add ERPTargetable interface and return true in CanBeTargetedBy.
  4. Bind events: Use UERPChannelListenerComponent set to the Target channel to listen to OnCandidateAcquired / OnCandidateLost events to drive UI/logic.

For custom channels or to handle multiple channels, use UERPChannelListenerComponent to mirror a single channel (acquired/lost) without cross-channel gameplay logic, or bind directly to the perception component's generic OnChannelCandidateAcquired / OnChannelCandidateLost events.

Cheat Sheetâ€‹

  • Lower score wins. Keep scorers cheap; aggregator/resolver rely on that convention.
  • Multi-channel architecture: Define channels like Target & Interaction. Each has its own Pipeline and best candidate.
  • Use channel listeners: Add UERPChannelListenerComponent and set its ChannelId to listen to a single channel's acquired/lost events.
  • Debug: enable bDebugPerception; console showdebug PERCEPTION.
  • Perf levers: reduce DefaultSamplingRange, set TickInterval, tighten Sampler filters.

Events & Debugâ€‹

Debugging Perceptionâ€‹

Enable bDebugPerception on UERPPerceptionComponent to log winner/candidates.

Generic Channel Events (multi-channel)â€‹

If you use custom channels (beyond Target / Interaction), you can consume perception state in a gameplay-agnostic way:

  • OnChannelCandidateAcquired(FName ChannelId, AActor* Candidate)
  • OnChannelCandidateLost(FName ChannelId, AActor* Candidate)

To avoid cross-channel logic in your gameplay code, prefer adding UERPChannelListenerComponent and set its ChannelId. It mirrors a single channel and exposes:

  • OnCandidateAcquired(AActor*)
  • OnCandidateLost(AActor*)
  • GetCurrentCandidate() / HasCandidate()
