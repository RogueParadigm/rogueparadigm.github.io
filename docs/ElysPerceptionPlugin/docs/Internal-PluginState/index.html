Internal Plugin State

Note: This is an internal technical state document for plugin developers and maintainers. For user-facing documentation, see the Documentation README.

Date: 2026-01-15

This document is a snapshot of the plugin's current architecture, features, and file roles.

Private/Core/ERPPerceptionComponent.cppâ€‹

Implements the runtime behavior:

  • Continuous evaluation (tick)
  • Per-channel pipeline execution (sample â†’ filter â†’ score â†’ select)
  • Multi-channel state tracking (CurrentBestByChannel, PrevBestByChannel)
  • Change detection and event emission (acquired/lost per channel)
  • Perception context building (aim ray, screen reference, viewport info)

The component is pure perception only:

  • No cross-channel priority logic
  • No winner/focus semantics
  • No lock state
  • No gameplay actions

3) Architecture overviewâ€‹

Core concept: Pure Perception, Multi-Channelâ€‹

A single component (UERPPerceptionComponent) evaluates multiple independent channels every tick.

Each channel:

  • Has a unique FName identifier (e.g., "Combat", "Interaction", "Ally")
  • Executes its own pipeline: Sample â†’ Filter â†’ Score â†’ Select
  • Tracks its own best candidate independently
  • Emits generic events when its candidate changes

No cross-channel logic:

  • No winner/focus/priority between channels
  • No lock state
  • Each channel is completely independent

Multi-channel state managementâ€‹

Internal state is stored in two parallel maps:

TMap<FName, TObjectPtr<AActor>> CurrentBestByChannel;
TMap<FName, TObjectPtr<AActor>> PrevBestByChannel;

Every tick:

  1. For each configured channel in ChannelPipelines:
    • Execute the channel's pipeline to find the best candidate
    • Compare with CurrentBestByChannel[ChannelId]
    • If changed: emit events and update maps
  2. PrevBestByChannel preserves the previous frame's state for change detection

Event flow per channel:

  • If new candidate differs from current: emit OnChannelCandidateLost (if previous existed), then OnChannelCandidateAcquired (if new exists)
  • Each event passes (FName ChannelId, AActor* Candidate)

4) Pipeline execution (runtime details)â€‹

Channel pipeline architectureâ€‹

Each channel in ChannelPipelines defines:

  • Sampler â€” Collects candidate actors from the world (e.g., sphere overlap, box trace)
  • Filters â€” Array of filters to validate/reject candidates (e.g., interface checks, line-of-sight)
  • Scorer â€” Assigns a numeric score to each candidate (e.g., distance, screen-space proximity)
  • Selector â€” Chooses the best candidate from scored results (e.g., lowest score, highest score)

Evaluation flow per channelâ€‹

  1. Sample: Sampler produces a list of actor candidates
  2. Filter: Each filter validates candidates (actors failing any filter are rejected)
  3. Score: Scorer assigns a numeric value to each remaining candidate
  4. Select: Selector picks the best candidate based on scoring strategy
  5. Compare: Result is compared with CurrentBestByChannel[ChannelId]
  6. Emit: If changed, emit OnChannelCandidateLost and/or OnChannelCandidateAcquired

Perception contextâ€‹

The component builds a FERPPerceptionContext struct passed to all pipeline stages:

  • Origin & Direction: From GetPerceptionAimRay() (default: owner view point)
  • Screen Reference: From GetScreenReferenceNdc() (optional, for screen-space scorers)
  • Viewport Size: If bProvideViewportInfoWhenAvailable is true (for local PlayerController)
  • Sampling Range: From DefaultSamplingRange (samplers may ignore or override)

Extensibilityâ€‹

  • Override GetPerceptionAimRay in BP/C++ for custom camera setup
  • Override GetScreenReferenceNdc for custom screen-space reference (e.g., mouse cursor)
  • Create custom samplers, filters, scorers, and selectors as needed
  • Define any number of channels with unique pipeline configurations

5) Events: rules and expectationsâ€‹

Generic multi-channel eventsâ€‹

The component emits only two event types, both generic and channel-agnostic:

OnChannelCandidateAcquired(FName ChannelId, AActor Candidate)*

  • Fired when a channel acquires a new best candidate
  • ChannelId identifies which channel changed
  • Candidate is the new best actor (never null)

OnChannelCandidateLost(FName ChannelId, AActor Candidate)*

  • Fired when a channel loses its previous best candidate
  • ChannelId identifies which channel changed
  • Candidate is the actor that was lost (never null)

Event orderingâ€‹

When a channel's best candidate changes from ActorA to ActorB:

  1. OnChannelCandidateLost(ChannelId, ActorA) fires first
  2. OnChannelCandidateAcquired(ChannelId, ActorB) fires second

When a channel's best candidate changes from ActorA to none:

  1. OnChannelCandidateLost(ChannelId, ActorA) fires
  2. No acquisition event (no new candidate)

When a channel's best candidate changes from none to ActorA:

  1. No loss event (no previous candidate)
  2. OnChannelCandidateAcquired(ChannelId, ActorA) fires

No focus/winner semanticsâ€‹

Unlike legacy systems, these events carry no priority/focus information:

  • No bHasCurrentFocus parameter
  • No focus-status-changed events
  • No winner code

If cross-channel priority is needed, implement it in gameplay code by listening to multiple channels.

6) Feature set recapâ€‹

Pure perception (channel-agnostic)â€‹

  • Multi-channel evaluation via configurable pipelines
  • Independent best candidate tracking per channel
  • Generic events (acquired/lost) with channel identification
  • No built-in gameplay semantics (no target/interaction/focus/lock)

Perception context providersâ€‹

  • Pluggable aim ray source (GetPerceptionAimRay)
  • Optional screen-space reference (GetScreenReferenceNdc)
  • Viewport info passthrough for screen-space scorers

Pipeline extensibilityâ€‹

  • Custom samplers (how to find candidates)
  • Custom filters (how to validate candidates)
  • Custom scorers (how to rank candidates)
  • Custom selectors (how to choose the best)

Configuration flexibilityâ€‹

  • Define any number of channels with unique FName identifiers
  • Each channel has independent pipeline configuration
  • No hardcoded channel types or priorities

7) Non-goals (intentionally excluded)â€‹

  • No cross-channel priority/winner logic
  • No focus semantics
  • No lock state or target freezing
  • No UI prompts or input binding
  • No gameplay orchestration or action execution
  • No GAS dependencies
  • No hardcoded channel types (e.g., "Target", "Interaction")
