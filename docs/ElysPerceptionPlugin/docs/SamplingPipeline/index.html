Sampling Pipeline Guide

This guide explains how the ElysPerceptionPlugin's sampling pipeline works under the hood.

Overviewâ€‹

What is the Pipeline?â€‹

The Pipeline is the core detection and evaluation system that finds and ranks potential targets or interactables in your game world.

Think of it like a factory assembly line:

  1. Gather potential objects (Sampler)
  2. Remove invalid ones (Filters)
  3. Rate the remaining ones (Scorers)
  4. Combine ratings (Aggregator)
  5. Pick the best one (Resolver)

Why Use a Pipeline?â€‹

Benefits:

  • Modular: Each stage does one job
  • Configurable: Mix and match components without code
  • Extensible: Add custom logic easily
  • Reusable: Same components work for different use cases
  • Maintainable: Debug one stage at a time

Comparison to traditional systems:

TraditionalPipeline
Hardcoded logicConfigurable components
One-size-fits-allMix and match
Hard to customizeEasy to extend
Tightly coupledLoosely coupled

How It Worksâ€‹

The Big Pictureâ€‹

Every tick, the ERPPerceptionComponent:

  1. Builds Context - Snapshot of player position, view direction, etc.
  2. Runs Each Channel - Target, Interaction, Custom channels run independently
  3. Determines Focus - Which channel's result gets priority
  4. Fires Events - Notifies your game of changes

A Simple Exampleâ€‹

Let's trace through a target detection:

Setup:

  • Player at position (0, 0, 0)
  • Enemy A at (500, 0, 0) - 500 units away
  • Enemy B at (1000, 0, 0) - 1000 units away
  • Both enemies implement IERPTargetable

Pipeline Execution:

  1. Sampler runs:

    • Sphere overlap finds both Enemy A and Enemy B
    • Result: [Enemy A, Enemy B]

  2. Filters run:

    • ERPTargetableFilter checks if each enemy can be targeted
    • Enemy A: CanBeTargetedBy() â†’ true âœ“
    • Enemy B: CanBeTargetedBy() â†’ true âœ“
    • Result: [Enemy A, Enemy B] (both pass)

  3. Scorers run:

    • ERPDistanceScorer calculates distance for each
    • Enemy A: 500 units â†’ score 0.25 (normalized)
    • Enemy B: 1000 units â†’ score 0.5 (normalized)
    • Result: [(Enemy A, 0.25), (Enemy B, 0.5)]

  4. Aggregator runs:

    • ERPIdentityAggregator passes through scores (only one scorer)
    • Result: [(Enemy A, 0.25), (Enemy B, 0.5)]

  5. Resolver runs:

    • ERPLowestScoreResolver picks the lowest score
    • Result: Enemy A wins (0.25 < 0.5)

  6. Events fire:

    • OnTargetAcquired(Enemy A, true) fires
    • Your UI shows target indicator on Enemy A

Pipeline Stagesâ€‹

Stage 1: Samplerâ€‹

Job: Collect potential candidates from the world

How: Performs a spatial query (overlap, raycast, etc.)

Output: Array of actors [Actor1, Actor2, ...]

Default Implementation:

  • ERPSphereOverlapSampler - Sphere overlap around player

Configuration:

Sampler: ERPSphereOverlapSampler
  - Trace Channel: Pawn
  - Actor Class: Character
  - Ignore Context Actor: âœ“

What it does:

  • Performs SphereOverlapActors at player position
  • Uses configured trace channel for filtering
  • Optionally filters by actor class
  • Can ignore the player actor itself

Stage 2: Filtersâ€‹

Job: Remove invalid candidates (binary pass/fail)

How: Each filter checks one condition; ALL must pass

Output: Filtered array of actors

Default Implementations:

  • ERPTargetableFilter - Requires IERPTargetable interface
  • ERPInteractableFilter - Requires IERPInteractable interface

Configuration:

Filters:
  - ERPTargetableFilter
  - [Your custom filter]

Execution: Filters run in sequence with short-circuit evaluation:

  • If Filter 1 fails â†’ actor rejected immediately
  • If Filter 1 passes â†’ Filter 2 runs
  • And so on...

Example Filter Logic:

bool Passes(Context, Candidate)
{
    // Check if actor implements interface
    if (!Candidate->Implements<UERPTargetable>())
        return false;
    
    // Ask the actor if it can be targeted
    return IERPTargetable::Execute_CanBeTargetedBy(Candidate, Context.ContextActor);
}

Stage 3: Scorersâ€‹

Job: Assign numerical scores to candidates

How: Each scorer evaluates one aspect; produces one number per candidate

Output: Array of scores per candidate [[0.5, 0.3], [0.8, 0.2]]

Default Implementations:

  • ERPDistanceScorer - Score by distance (lower = closer)
  • ERPConeScorer - Score by angle from view direction
  • ERPViewportEllipseScorer - Score by screen position

Configuration:

Scorers:
  - ERPDistanceScorer (Normalize By Range: âœ“)
  - ERPConeScorer

What they do:

  • Distance Scorer: Closer objects score lower (better)
  • Cone Scorer: Objects in center of view score lower
  • Viewport Scorer: Objects near screen center score lower

Score Ranges:

  • Typically normalized to 0.0 - 1.0
  • Lower scores = better (by convention)
  • Can be any range, but normalize for easier combination

Stage 4: Aggregatorâ€‹

Job: Combine multiple scores into one final score

How: Takes array of scores, returns single number

Output: One score per candidate

Default Implementations:

  • ERPIdentityAggregator - Pass through first score (for single scorer)
  • ERPAdditiveAggregator - Sum all scores
  • ERPMultiplicativeAggregator - Multiply all scores

Configuration:

Aggregator: ERPIdentityAggregator  // For 1 scorer
or
Aggregator: ERPAdditiveAggregator  // For multiple scorers

Example:

Candidate A scores: [0.3 (distance), 0.2 (angle)]
Candidate B scores: [0.5 (distance), 0.1 (angle)]

Additive Aggregator:
  A: 0.3 + 0.2 = 0.5
  B: 0.5 + 0.1 = 0.6
  â†’ A wins (lower score)

Stage 5: Resolverâ€‹

Job: Select the winning candidate

How: Examines all final scores and picks the best

Output: Single best candidate (or none)

Default Implementations:

  • ERPLowestScoreResolver - Lowest score wins
  • ERPHighestScoreResolver - Highest score wins

Configuration:

Resolver: ERPLowestScoreResolver  // For distance-based (closer = better)
or
Resolver: ERPHighestScoreResolver  // For priority-based (higher = better)

What it does:

  • Iterates through all scored candidates
  • Finds minimum (or maximum) score
  • Returns that candidate as the winner

Context Systemâ€‹

What is Context?â€‹

The Context is a snapshot of the player's state at the moment of evaluation. It's built once per tick and shared across all pipeline stages.

Why?

  • Consistency: All stages see the same data
  • Performance: Avoid repeated queries (GetActorLocation, etc.)
  • Flexibility: Works with different owner types (Controller, Pawn, etc.)

Context Contentsâ€‹

struct FERPPerceptionContext
{
    // Core data (always present)
    AActor* ContextActor;           // The owner (player)
    FVector Origin;                 // Player position
    FVector Forward;                // Player facing direction
    
    // Optional aim ray (for camera/weapon aiming)
    bool bHasAimRay;
    FVector AimOrigin;
    FVector AimDirection;
    
    // Optional viewport data (for screen-space scoring)
    bool bHasViewportProjection;
    FVector2D ViewportCenterPx;
    float ViewportEllipseHalfWidthPx;
    float ViewportEllipseHalfHeightPx;
};

How Context is Builtâ€‹

  1. Get Owner: Component owner (PlayerController or Pawn)
  2. Get View Point: Position and rotation for perception
  3. Extract Aim Ray: Camera or weapon aim (if available)
  4. Calculate Viewport: Screen-space data (for PlayerController only)

Using Context in Pipeline Stagesâ€‹

Distance Scorer Example:

float Score(const FERPPerceptionContext& Context, AActor* Candidate)
{
    FVector CandidateLocation = Candidate->GetActorLocation();
    float Distance = FVector::Dist(Context.Origin, CandidateLocation);
    return Distance / MaxRange;  // Normalized
}

Cone Scorer Example:

float Score(const FERPPerceptionContext& Context, AActor* Candidate)
{
    FVector ToCandidate = (Candidate->GetActorLocation() - Context.Origin).GetSafeNormal();
    float DotProduct = FVector::DotProduct(Context.Forward, ToCandidate);
    return (1.0f - DotProduct) / 2.0f;  // Convert -1..1 to 0..1
}

Multi-Channel Supportâ€‹

What are Channels?â€‹

Channels are independent pipelines that run in parallel. Each channel maintains its own best candidate.

Common channels:

  • Target: Combat targeting (enemies)
  • Interaction: Object interaction (doors, items)
  • Custom: Healing, quest objectives, etc.

How Channels Workâ€‹

Each channel has:

  • Its own pipeline configuration (Sampler, Filters, Scorers, etc.)
  • Its own best candidate
  • Its own events

Channels run:

  • Independently (Target doesn't affect Interaction)
  • Simultaneously (all channels evaluated each tick)
  • With shared context (same snapshot)

Focus Systemâ€‹

When multiple channels have candidates, the system determines focus:

Focus rules:

  • Only ONE channel can have focus at a time
  • Focus determines which candidate is "primary"
  • Non-focus channels still track their best, but are "secondary"

Example:

Target Channel: Enemy A (has focus) â†’ Primary target, orange outline
Interaction Channel: Door B (no focus) â†’ Secondary, blue outline

Channel Configurationâ€‹

Example: Two channels

Target Pipeline:
  Channel Id: Target
  Range: 3000
  Sampler: Sphere (Pawn collision)
  Filters: TargetableFilter
  Scorers: DistanceScorer
  
Interaction Pipeline:
  Channel Id: Interaction
  Range: 300
  Sampler: Sphere (Visibility collision)
  Filters: InteractableFilter
  Scorers: DistanceScorer

Execution Flowâ€‹

Complete Tick Flowâ€‹

1. UERPPerceptionComponent::TickComponent()
   â†“
2. BuildContext()
   â†’ Get owner actor
   â†’ Get view point (position, rotation)
   â†’ Extract aim ray
   â†’ Calculate viewport data
   â†’ Create FERPPerceptionContext
   â†“
3. For each channel:
   â†“
   3a. Execute Pipeline:
       â†’ Sampler: Gather candidates
       â†’ Filters: Reject invalid
       â†’ Scorers: Assign scores
       â†’ Aggregator: Combine scores
       â†’ Resolver: Select best
   â†“
   3b. Store result (best candidate + score)
   â†“
4. Determine Focus Winner
   â†’ Compare channel priorities
   â†’ Assign focus to one channel
   â†“
5. Fire Events:
   â†’ OnCandidateAcquired (for new candidates)
   â†’ OnCandidateLost (for lost candidates)
   â†’ OnFocusStatusChanged (for focus changes)
   â†“
6. Update Debug Visualization (if enabled)

Performance Characteristicsâ€‹

Per Tick:

  • 1x Context build (cheap)
  • NÃ— Pipeline executions (one per channel)
  • MÃ— Candidate evaluations (depends on sampler results)

Optimization tips:

  • Keep sampling range reasonable
  • Use filters to reject early
  • Keep scorer logic lightweight
  • Adjust component tick interval if real-time isn't needed
